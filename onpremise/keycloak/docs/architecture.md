# Keycloak SSO architecture — auth flow and migration paths

This doc explains **why we put Keycloak in front of the apps**, **what the auth flow looks like before vs after**, and **what changes when we later add LDAP**. Read it as a reference before running the Phase 4-6 cutovers, or to onboard someone joining mid-rollout.

<br/>

## Before — each app talks to GitLab directly

```
user → ArgoCD/Harbor/vaultwarden → "Login with GitLab"
                                          │
                                          ▼
                                   GitLab (gitlab.example.com)
                                          │ OAuth2
                                          ▼
                                   ID token (issuer = gitlab.example.com)
                                          │
                                          ▼
                                   each app verifies the token → in
```

- A separate GitLab Application per app.
- Each app fetches the `groups` claim straight from the GitLab API.
- **Auth policies (MFA, session expiry, additional IdPs) are defined inside GitLab** — apps cannot extend them.

<br/>

## After — Keycloak brokering (post Phase 4-6 cutover)

```
user → ArgoCD/Harbor/vaultwarden → "Login with Keycloak"
                                          │
                                          ▼
                                   Keycloak (auth.example.com)
                                   `example` realm login page
                                          │
                              (no live SSO session)
                                          ▼
                                   click "Sign in with GitLab"
                                          │
                                          ▼
                                   GitLab (gitlab.example.com) login
                                          │
                                          ▼
                                   Keycloak federates the user
                                          │ (first time: realm DB row created)
                                          │ (server-group-map: hardcoded into /server)
                                          ▼
                                   Keycloak ID token (issuer = auth.example.com)
                                          │ groups: ["server"]
                                          ▼
                                   each app verifies the token → in
```

Key differences:

1. **The token issuer is now Keycloak.** Each app trusts `auth.example.com/realms/example`. GitLab's name no longer appears in app config.
2. **A live Keycloak SSO session bypasses the GitLab step** — sign in to one app, the rest go through silently (real SSO).
3. **First brokered login imports the user into Keycloak.** They show up in the `example` realm's Users list as `firstUser@example.com`, etc. GitLab-side changes flow back via federation refresh on the IdP `syncMode=IMPORT` policy.
4. **The `groups` claim is generated by Keycloak**, not GitLab. The IdP-level mapper `server-group-map` hardcodes every brokered user into `/server`, so existing ArgoCD `g, server, role:server-admin` rules and Harbor group filters keep working without policy edits.

<br/>

## What gets better

| Topic | Before | After |
|---|---|---|
| Adding another IdP (Google, MFA) | Configure on every app | One change in the Keycloak realm, all apps inherit |
| LDAP federation (later track) | Each app would integrate LDAP itself | Single Keycloak User Federation entry |
| Sessions / unified logout | Per-app | Keycloak Account Console |
| User source-of-truth | GitLab (only option) | Keycloak realm abstracts the IdP — GitLab becomes one of N backends |
| Number of GitLab Applications | One per app | Collapsible to a single brokering app (Phase 8) |

<br/>

## Costs

- **One extra click** when the Keycloak SSO session has expired — but `kc_idp_hint=gitlab` as a query param can auto-redirect to GitLab, removing the click during cutover.
- **The OIDC `sub` value changes.** A token issued via direct GitLab and a token via Keycloak brokering have different `sub` strings. Harbor is the one most affected — existing OIDC users need their project memberships re-mapped (admin procedure in [harbor-migration-en.md](harbor-migration-en.md)).
- **One more component to operate** — Keycloak + its dedicated PostgreSQL pod.

<br/>

## What Phase 4-6 actually changes in each app

Only three OIDC fields differ — everything else stays the same:

| Field | Before | After (Phase 4-6) |
|---|---|---|
| issuer | `http://gitlab.example.com` | `https://auth.example.com/realms/example` |
| clientId | GitLab Application ID | `harbor` / `vaultwarden` / `argocd` |
| clientSecret | GitLab Application Secret | The Phase 3 client secret recovered from Keycloak |

**This change holds whether Keycloak's backend is GitLab or LDAP.** The app only sees the issuer (`auth.example.com`) — it doesn't know whether the realm brokers to GitLab or federates to LDAP.

```
[before]                                   [after Phase 4-6]                        [after LDAP]
app → GitLab                               app → Keycloak → GitLab                  app → Keycloak → LDAP
                                           (one-time Phase 4-6 change)              (Keycloak-internal change, app untouched)
```

<br/>

## What changes when LDAP is introduced

Covered by the follow-up plan ([openldap-keycloak-federation.md](../../../.claude/plans/gitlab-project-kuberntes-infra-openldap-keycloak-federation.md)):

1. Install OpenLDAP (slapd).
2. In the Keycloak `example` realm → User Federation → add an LDAP provider (URL, baseDN, bind creds, sync options).
3. Add a group-ldap-mapper so the LDAP `ou=Groups` entries `server` / `global-admin` map onto Keycloak groups.
4. (Optional) Disable the GitLab IdP — make LDAP the sole source-of-truth.
5. Users sign in with LDAP credentials on their next session (or pick GitLab IdP if it's still enabled).

**Zero change to the apps.** ArgoCD `policy.csv`, Harbor group filter, vaultwarden `sso.authority` — all stay as written by Phase 4-6.

<br/>

## User impact during the Phase 4-6 cutover

| App | User impact | Notes |
|---|---|---|
| **ArgoCD** | Sign in once via the new flow | RBAC keys off `email`
| **vaultwarden** | Sign in once via the new flow | `signupsMatchEmail: true` keeps email-based matching |
| **Harbor** | Sign in once + a one-time admin task | OIDC `sub` changes → existing project memberships need to be re-attached. Admin runs the procedure in [harbor-migration-en.md](harbor-migration-en.md); end-users just sign in afresh |

Bottom line: **no re-registration, no re-permissioning, no new passwords.** Same GitLab account, one fresh sign-in.

<br/>

## Migration path comparison

If LDAP is the long-term endpoint, do we even need the GitLab brokering step? Three options:

### Scenario A — GitLab brokering → stabilize → add LDAP (current plan, recommended)

- Pros: Existing GitLab users go through **zero re-registration, one re-login**. Minimal user impact.
- Cost: One extra GitLab Application (already created — `Keycloak Brokering (example)`).
- Flow: Phase 4-6 cutover → soak → follow-up plan adds LDAP federation → disable GitLab IdP (Phase 8 or later cleanup).

### Scenario B — Skip GitLab brokering, jump straight to LDAP

- Pros: No intermediate scaffolding.
- Cost: **Every ArgoCD/Harbor/vaultwarden user has to be created from scratch in LDAP.** Cutover means re-registration *and* re-login, plus permission re-mapping.
- Bigger problem: LDAP (OpenLDAP) lands later than Keycloak does. Until LDAP is up, auth either breaks or has to keep relying on GitLab.

### Scenario C — LDAP first (defer Phase 4-7 → install LDAP → run Phase 4-7)

- Pros: LDAP is already source-of-truth before the app cutover starts.
- Cost: Apps stay on direct GitLab during the LDAP build-out (Keycloak sits idle). Cutover and LDAP migration collide on the same change window — higher cutover risk.

<br/>

## Conclusion

**Scenario A (the current plan) is the recommended path** — because:

1. Phase 3 (GitLab IdP registration) is **already done and almost free**. Just one extra GitLab Application.
2. Phase 4-6 (rewriting each app's OIDC) has to happen once either way — A and B share that cost.
3. **Zero user impact** — existing GitLab accounts keep working. LDAP rollout stays on a separate, lower-risk track.
4. After LDAP lands, the GitLab IdP can be disabled with a single toggle.
